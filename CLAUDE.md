# CLAUDE.md

## Project Overview

**eSimplu** is a platform for the Romanian/Moldovan diaspora in Europe. It provides useful articles, a directory of services (accountants, lawyers, shops...), a marketplace for Moldovan/Romanian producers, and a delivery platform (point-to-point logistics).

The site targets 4 countries: France, Germany, Italy, UK. Content is in Romanian. The country is selected by the user (not in the URL) and stored in a cookie.

## Commands

```bash
# Local development
docker compose up -d          # Start PostgreSQL
npm run dev                   # Start Next.js dev server (localhost:3000)
npx prisma studio             # Database GUI (inspect data)
npx prisma migrate dev        # Run database migrations
npx prisma generate           # Regenerate Prisma client after schema changes

# Database
npm run db:seed               # Seed articles + 86 real services (RO/MD)
npm run db:reset              # ⚠ Reset DB + reseed (drop toutes les données)

# Drafts (claim flow)
npm run service:draft -- \
  --title "..." --category mutari-transport --city Paris --country fr \
  --phone "+33..." --description "..." [--whatsapp ... --languages ro,fr]
# → crée un ServiceListing + Invitation, imprime l'URL /claim/<token>

# Testing
npm test                      # Run tests
npm run lint                  # Run ESLint

# Build
npm run build                 # Production build
```

### Notes Prisma + Claude

`prisma migrate dev` exige un TTY interactif et `prisma migrate reset` exige
`PRISMA_USER_CONSENT_FOR_DANGEROUS_AI_ACTION` quand l'agent l'invoque. Pour
générer une migration en mode non-interactif :

```bash
ts=$(date +%Y%m%d%H%M%S) && d="prisma/migrations/${ts}_<name>" && mkdir -p "$d"
npx prisma migrate diff --from-config-datasource --to-schema prisma/schema.prisma --script \
  | sed '/^◇ injected/d;/^Loaded Prisma config/d' > "$d/migration.sql"
npx prisma migrate deploy
npx prisma generate
```

Après tout changement de schéma, **redémarrer le dev server** (`pkill -9 -f "next dev"; rm -rf .next; npm run dev`) — Turbopack cache les chunks SSR et part en ENOENT sinon.

## Key Design Decisions

- **No WordPress** — content (articles) is generated by AI via scripts, not a CMS
- **No monorepo/workspaces** — single package.json at root, everything in one app
- **Country = data filter, not URL** — user selects country via modal/selector, stored in cookie. No `/fr/`, `/de/` URL segments
- **Auth.js JWT strategy** — Google + Credentials (email/password bcrypt) actifs, Facebook en attente. Prisma adapter en DB pour les comptes OAuth.
- **PostgreSQL, not MySQL** — JSONB support, better ecosystem with Prisma/Next.js
- **Caddy, not Nginx** — HTTPS automatique, config minimale, pas de Cloudflare
- **Services, Marketplace, Delivery are independent modules** — linked only via userId, no shared Provider model
- **Polymorphic `Invitation` model (PR #25)** — un seul modèle gère le claim flow pour Service, Trip, Product. La route `/claim/[token]` dispatche selon `targetType`. Pas de FK Prisma directe sur la cible (cascade applicative).
- **Public visibility = NOT REJECTED** — `ServiceStatus` est réduit à `PENDING|REJECTED`. Une fiche pré-remplie avec une `Invitation` PENDING est visible sur `/services` dès création ; refus du propriétaire ou expiration → soft-delete (REJECTED).

## Country Logic

- First visit: modal asks user to choose country (FR, DE, IT, UK)
- Cookie `country` stores the choice (1 year)
- Articles and Services are filtered by `countries[]` field
- Marketplace products filtered by seller's `deliveriesTo[]`
- Delivery trips: no auto-filter, user searches by origin/destination
- `?country=fr` query param overrides cookie for shareable links

## Deployment

**VPS (production) :**
- App directory: `~/esimplu` (NOT `/var/www/esimplu`)
- PM2 process: `esimplu` (NOT `esimplu-web`)
- Secrets: `~/ecosystem.config.js` (chmod 600) — pas de `.env.local` en prod
- Deploy manuel: `cd ~/esimplu && git pull && npm install && npx prisma migrate deploy && npm run build && pm2 restart esimplu`
- Pour changer des env vars: `pm2 stop esimplu && pm2 delete esimplu && pm2 start ~/ecosystem.config.js && pm2 save`

**GitHub Actions (`deploy.yml`)** — corrigé (chemins et nom PM2). Pour activer le CI/CD auto, il reste à configurer les GitHub Secrets : `VPS_HOST`, `VPS_USER`, `VPS_SSH_KEY`.

## Git Workflow

**Branches :**
- `main` — branche de production, toujours déployable
- `feat/*` — nouvelles fonctionnalités
- `fix/*` — corrections de bugs
- `chore/*` — tooling, scripts npm, refactos sans changement fonctionnel
- `docs/*` — docs uniquement

**Règles :**
- Ne jamais push directement sur `main` — toujours passer par une branche + PR
- `git pull origin main` avant de créer une branche
- Nommer les commits en anglais avec un préfixe : `feat:`, `fix:`, `refactor:`, `docs:`, `ci:`, `chore:`
- Faire une PR sur GitHub pour review avant de merger dans `main`
- Quand une PR est basée sur une autre PR (chaînée), **changer la base** avec `gh pr edit <N> --base <autre-branche>` pour que le diff GitHub reste propre

**Branches actives :** voir le tableau dans `ROADMAP.md` (mis à jour à chaque PR ouverte).

## Collaboration

L'équipe = 2 fondateurs, chacun travaille avec son IA en autonomie. Pas de mentor mode actif. Communication en français côté UI / commits, anglais côté code et messages de commit (préfixe conventional commits).

Tout contenu utilisateur (pages, formulaires, emails, messages d'erreur visibles) doit être en **roumain** — c'est la langue de la diaspora cible.
