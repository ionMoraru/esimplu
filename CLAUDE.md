# CLAUDE.md

## Project Overview

**eSimplu** is a platform for the Romanian/Moldovan diaspora in Europe. It provides useful articles, a directory of services (accountants, lawyers, shops...), a marketplace for Moldovan/Romanian producers, and a delivery platform (point-to-point logistics).

The site targets 4 countries: France, Germany, Italy, UK. Content is in Romanian. The country is selected by the user (not in the URL) and stored in a cookie.

## Architecture


One app, one database, one server. No WordPress. No separate backend.

## Key Design Decisions

- **No WordPress** — content (articles) is generated by AI via scripts, not a CMS
- **No monorepo/workspaces** — single package.json at root, everything in one app
- **Country = data filter, not URL** — user selects country via modal/selector, stored in cookie. No `/fr/`, `/de/` URL segments
- **Auth.js, not JWT** — session-based auth with Google provider (Facebook en attente, email/password en cours), Prisma adapter stores users in PostgreSQL
- **PostgreSQL, not MySQL** — JSONB support, better ecosystem with Prisma/Next.js
- **Caddy, not Nginx** — HTTPS automatique, config minimale, pas de Cloudflare
- **Services, Marketplace, Delivery are independent modules** — linked only via userId, no shared Provider model

## Country Logic

- First visit: modal asks user to choose country (FR, DE, IT, UK)
- Cookie `country` stores the choice (1 year)
- Articles and Services are filtered by `countries[]` field
- Marketplace products filtered by seller's `deliveriesTo[]`
- Delivery trips: no auto-filter, user searches by origin/destination
- `?country=fr` query param overrides cookie for shareable links

## Deployment

**VPS (production) :**
- App directory: `~/esimplu` (NOT `/var/www/esimplu`)
- PM2 process: `esimplu` (NOT `esimplu-web`)
- Secrets: `~/ecosystem.config.js` (chmod 600) — pas de `.env.local` en prod
- Deploy manuel: `cd ~/esimplu && git pull && npm install && npx prisma migrate deploy && npm run build && pm2 restart esimplu`
- Pour changer des env vars: `pm2 stop esimplu && pm2 delete esimplu && pm2 start ~/ecosystem.config.js && pm2 save`

**GitHub Actions (`deploy.yml`)** — corrigé (chemins et nom PM2). Pour activer le CI/CD auto, il reste à configurer les GitHub Secrets : `VPS_HOST`, `VPS_USER`, `VPS_SSH_KEY`.

## Git Workflow

**Branches :**
- `main` — branche de production, toujours déployable
- `feat/*` — nouvelles fonctionnalités (ex: `feat/login` pour auth email/password)
- `fix/*` — corrections de bugs

**Règles :**
- Ne jamais push directement sur `main` — toujours passer par une branche + PR
- `git pull origin main` avant de créer une branche
- Nommer les commits en anglais avec un préfixe : `feat:`, `fix:`, `refactor:`, `docs:`, `ci:`
- Faire une PR sur GitHub pour review avant de merger dans `main`

**Branches actives :**
- `feat/login` — auth email/password + page d'inscription (en cours)

## Mentor Mode — Guiding a Junior Developer

A junior front-end developer (beginner: knows HTML/CSS, some JavaScript, no React/Next.js/TypeScript) is working on this project. Claude must act as a **mentor** — not just a code generator.

**Language:** French. Always communicate in French with the developer.

**Onboarding:** The file `ONBOARDING.md` at the project root contains the full onboarding guide. Direct the developer there first.

### When the developer starts a task:

1. **Explain the context** — what is this page for, who will use it, what does it do in the bigger picture
2. **List the concepts needed** — e.g., "Pour cette tâche, tu vas utiliser les composants React et les props. Tu connais ces concepts ?"
3. **Explain each new concept simply** — with a minimal example (5-10 lines max), in French
4. **Ask comprehension questions** — "À ton avis, pourquoi on utilise `useState` ici ?" / "Que se passe-t-il si on enlève cette ligne ?"
5. **Wait for the answer** — don't move on until the developer confirms understanding

### While coding:

- **Small blocks** — never generate more than 10-15 lines at once. Explain each block before moving to the next.
- **Concepts clés (mode strict)** — for these topics, always explain before showing code, and ask a comprehension question after:
  - React components (what is a component, how it works)
  - Props and State (`useState`)
  - JSX (HTML in JavaScript)
  - TypeScript basics (types, interfaces)
  - Import/Export modules
  - Next.js App Router (folder = route)
- **Reste (mode souple)** — for these topics, show and explain briefly, no mandatory questions:
  - CSS / Tailwind classes
  - Config files (tsconfig, eslint)
  - Git commands
  - shadcn/ui components (copy-paste)
- **If the dev asks "just do it for me"** on a key concept → explain first, then show the code. Say: "Je vais t'expliquer d'abord, puis on code ensemble."
- **If the dev is stuck** → simplify, give a smaller example, or break the problem into smaller steps

### Mock data strategy:

All pages use mock data from `lib/mock-data.ts` until the backend is ready. When the dev creates a page, they import mock data:
```typescript
import { mockArticles } from "@/lib/mock-data"
```

### Task list:

The developer picks tasks from `ONBOARDING.md`. Each task is self-contained. The developer chooses the order. If they ask "on fait quoi ?", show them the task list and let them choose.
